Open-source and self-hosted

A technical summary of what KiloLock is and is not.

KiloLock OSS is a self-hosted Terraform and OpenTofu HTTP backend. It stores state in PostgreSQL rather than as a flat state file, which makes query, history, drift, and some more advanced coordination workflows possible.

The intended low-risk adoption path is simple: use it as a backend first. The deeper capabilities exist, but they are not required to get value from the project.

Read the README Self-hosting guide

Short version

  • Works as a Terraform/OpenTofu HTTP backend.
  • Can be self-hosted with Docker Compose.
  • Adds queryable state and resource history.
  • Includes more advanced KiloLock-specific workflows.

Important framing

This page is meant to help technical users evaluate fit, not to present KiloLock as a finished replacement for every Terraform platform product.

Summary

What KiloLock OSS is

Core role

KiloLock OSS is the open-source backend and self-hosted core. It covers backend protocol handling, state storage, query workflows, refresh and drift workflows, repair-oriented tooling, and the server-side foundations for more advanced orchestration.

Adoption model

The practical first use is still ordinary: configure a standard HTTP backend, run terraform init, and keep using Terraform or OpenTofu. Users do not need to adopt the whole KiloLock model on day one.

Scope

What problem it is trying to solve

KiloLock is built around the idea that some infrastructure really does live as one tightly coupled system. In those cases, splitting into many smaller states does not always remove coordination pain. Sometimes it only moves that pain elsewhere.

The project is therefore not based on the assumption that every large Terraform state is a design failure. It assumes some large states are real, and tries to make them easier to inspect, operate, and coordinate safely.

Today

What KiloLock can do today

This is the practical, present-tense view of the OSS project.

HTTP backend compatibility

It works as a Terraform/OpenTofu HTTP backend, so the initial integration path stays familiar.

Self-hosting

It can be run locally or self-hosted with Docker Compose, including a fuller prod-like path.

Queryable state

State is stored in PostgreSQL in a way that supports direct queries over resources, dependencies, versions, and related metadata.

Refresh and drift workflows

It includes provider-aware refresh and drift-oriented workflows that do not depend on a full Terraform cycle every time.

History and repair workflows

It supports per-resource history inspection and targeted repair or rollback-oriented workflows.

Advanced coordination work

It also contains the foundations for scoped workflows, reservations, and more advanced coordination patterns when users adopt KiloLock-specific tooling.

Boundaries

What is true, but should not be oversold

Advanced features exist

Resource-aware coordination, scoped apply behavior, richer queryability, and repair workflows are part of the project direction and already exist in meaningful form.

But they are not the first promise

The first promise should stay modest: self-hosted backend compatibility and better state visibility. The deeper workflow changes are the second step, not the entrance requirement.

Limits

What KiloLock OSS does not claim to do

Not a full Terraform Cloud replacement

It is not presented as a complete hosted platform, runner product, or full SaaS control experience.

Not a fix for every Terraform performance problem

Storing state differently helps backend visibility and coordination, but it does not erase Terraform's own graph-build and provider execution costs.

Not magic parallelism

Parallel-safe workflows only make sense where resource footprints are genuinely disjoint. Conflicting changes still need coordination.

Not a substitute for sound engineering

It does not merge HCL for you, remove design coupling, or eliminate the need for careful infrastructure ownership.

Quickstart

Fastest way to evaluate it

The easiest way to judge fit is to run the self-hosted stack, point a backend at it, and see whether the basic HTTP backend path and state visibility match your expectations. 

cp .env.example .env
docker-compose up --build -d

# then configure backend "http"
terraform init
terraform apply

For the fuller operator-oriented setup, use the self-hosted bootstrap runbook.
