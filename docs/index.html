Self-hosted Terraform/OpenTofu state engine

Scale engineers, not state-file locks.

Kilolock replaces the flat .tfstate blob with a normalized PostgreSQL graph so teams can query state, inspect drift, repair a single resource, and coordinate parallel work on large shared state with much better visibility than a whole-workspace lock.

Read the quickstart See repair workflows

Main value

The headline benefit is not “SQL is neat.” It is letting more engineers work safely on the same large state without forcing everything through a single coarse lock.

Drop-in path

Existing Terraform and OpenTofu workflows still talk to an HTTP backend. Same HCL, same providers, same CLI habits — different engine under the state.

Self-hosted

This repository is the OSS core: kl, kld, klc, local compose, query/repair flows, and operator APIs.

Large state is not always the mistake

Sometimes the infrastructure really is tightly coupled. Forcing it into many smaller Terraform states can create a distributed monolith where the dependency graph still exists, but now it is harder to reason about and harder to change safely.

In those cases, a large shared state can be the honest representation of the system. Kilolock is built for that reality: better collaboration on one large state, better visibility, and better repair workflows when something goes wrong.

Why teams hit a wall with large state

Flat state file

  • Whole-workspace lock for unrelated changes
  • Slow refresh and plan loops as state grows
  • Weak answers to “what changed?” or “what drifted?”
  • Emergency repair usually means blunt whole-state recovery

Kilolock

  • Normalized resource graph in PostgreSQL
  • Reservation-aware orchestration for scoped work
  • Queryable resource history and drift
  • Resource-level inspection and repair workflows

Architecture at a glance

Client workflows stay API-only. Runtime and control services own the private databases.

Why parallel apply matters

The technical backend work matters because it removes a people bottleneck: more engineers can work on the same large state without queueing behind one lock.

What you get today

Queryable state

Run backend-native queries over normalized resources, dependencies, outputs, and drift.

Provider-aware refresh

Refresh state out of band and inspect drift without paying full plan refresh every time.

Scoped/orchestrated apply

Use file- and target-scoped workflows with reservation-aware orchestration.

Resource repair

Preview history and repair a single resource’s state bookkeeping without blunt whole-state rollback.

Control-plane APIs

Run self-hosted operator workflows for lifecycle, routing, entitlements, and control operations.

Parallel-apply foundations

The concurrency story is real: disjoint changes on large shared state are the center of the design.

Quickstart

docker-compose up --build -d

Then point Terraform’s backend "http" at http://localhost:8080/states/<workspace>/<environment>/<state>.

Why this matters operationally

  • parallel collaboration on one honest shared state
  • resource-level query, history, repair, and rollback
  • less pressure to invent artificial state splits just to escape locking

Good fit

  • Large shared states with multiple engineers
  • Teams that need better visibility into drift and blast radius
  • Operators who want self-hosted control over state data

Out of scope here

  • Hosted customer portal UX
  • Billing/signup glue and managed-service packaging
  • Cloud-specific internal deployment repo content
